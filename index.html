How to test each protocol

Testing REST APIs — curl, Postman, k6
REST is the easiest to test — any HTTP client works. curl: curl -X POST https://api.example.com/users -H "Content-Type: application/json" -d '{"name":"Alice"}'. Postman: import OpenAPI spec, generate requests automatically. k6 for load testing: import http from 'k6/http'; export default function() { http.get('https://api.example.com/users'); }. Integration testing: use supertest (Node.js) for in-process testing — no network call needed.

Testing GraphQL — GraphQL Playground, Apollo Studio, k6
GraphQL always uses POST to a single endpoint. GraphQL Playground / Apollo Sandbox: built-in schema introspection, query builder, autocomplete. curl: curl -X POST /graphql -H "Content-Type: application/json" -d '{"query":"{ users { id name } }"}'. Load testing with k6-xk6-browser: test query performance under load. Key test cases: N+1 query detection (use DataLoader), depth limiting (prevent deeply nested queries), introspection disabling in production.

Testing gRPC — grpcurl, evans, Postman
gRPC uses binary Protocol Buffers — can't use plain curl. grpcurl: grpcurl -d '{"user_id": "123"}' localhost:50051 users.UserService/GetUser. evans: interactive gRPC CLI with proto reflection. Postman (v9.7+): supports gRPC natively — import .proto file and test interactively. Load testing: ghz tool — ghz --call users.UserService/GetUser -d '{"user_id":"1"}' localhost:50051. Challenge: server reflection must be enabled for grpcurl without a .proto file.

FAQ — REST GraphQL gRPC testing
gRPC is the fastest — Protocol Buffers binary serialization is 3-10x more compact than JSON, and HTTP/2 multiplexing reduces connection overhead. Typical benchmark: gRPC ~12ms, REST ~45ms, GraphQL ~55ms for equivalent data. However, GraphQL can significantly reduce payload size vs REST by requesting only needed fields — on mobile networks, this often matters more than raw latency. For browser-facing APIs, REST and GraphQL are better choices due to gRPC's limited browser support.
grpcurl (like curl for gRPC): grpcurl -plaintext -d '{"user_id":"1"}' localhost:50051 users.UserService/GetUser. Requires server reflection enabled or a .proto file. evans: interactive REPL for gRPC testing. Postman (v9.7+): GUI gRPC client, import .proto. ghz for load testing. For Go: use google.golang.org/grpc/test/bufconn for in-process testing. For Node.js: use @grpc/grpc-js with mock server.
Choose GraphQL when: (1) Multiple client types need different data shapes (mobile vs desktop). (2) Over-fetching is a problem — REST returns 20 fields but mobile only needs 3. (3) Complex relationships — avoid 4 REST calls by fetching user + posts + comments + likes in one query. (4) Rapid frontend iteration — frontend can add fields without backend changes. Avoid GraphQL for: public APIs (REST is simpler), simple CRUD, caching-heavy architectures (REST caches naturally, GraphQL doesn't).
Yes — this is the most common production pattern. A typical architecture: Public API = REST (simple, widely supported). Frontend BFF = GraphQL (flexible queries from React/mobile). Internal microservices = gRPC (performance, streaming, type safety). Example: React app → GraphQL BFF → multiple gRPC microservices → public partner API via REST. The GraphQL layer acts as a translation layer that aggregates gRPC calls from multiple services into a single query.
